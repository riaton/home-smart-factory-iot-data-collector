# 実装コマンド

プランファイルを読み込み、タスクを順番に実装する。
タスク完了のたびにプランファイルの `[ ]` を `[x]` に更新する。

**コア原則**: バリデーション失敗は即座に修正する。壊れた状態を積み上げない。

**入力**: `$ARGUMENTS`

---

## 重要な行動ルール（CRITICAL BEHAVIORAL RULES）

以下のルールを厳守する。違反は失敗とみなす。

1. **チェックボックスを更新してから次へ進む。** タスクの `- [ ]` を `- [x]` に変更せずに次のタスクへ進んではならない。これは省略不可の必須手順である。
2. **テストが緑になってからチェックボックスを更新する。** GREEN 未達のタスクを `[x]` にしてはならない。
3. **次のタスク開始前に前のタスクの `[x]` を目視確認する。** プランファイルを読み直し、前タスクが `[x]` になっていることを確認してから次へ進む。
4. **全タスクが `[x]` になるまで静的解析・AI レビューフェーズに入らない。**
5. **失敗で停止する。** エラー・テスト失敗が発生したら黙って続行しない。停止してユーザーに報告する。

## フェーズ 0 — LOAD

プランファイルを読み込む:

```bash
cat "$ARGUMENTS"
```

ファイルが存在しない場合は停止:

```
エラー: プランファイルが見つかりません。
/plan <設計書パス> でプランを先に作成してください。
```

以下を取り出す:

- **タスクリスト** — `## タスクリスト` セクションの全行を読む
  - `- [x]` になっている行は完了済み（再開時はスキップ）
  - `- [ ]` になっている行が処理対象
- **タスク詳細** — 各タスクの IMPLEMENT / TEST / GOTCHA / RED / GREEN / REFACTOR
- **バリデーションコマンド** — `## バリデーションコマンド` セクション
- **受け入れ条件** — `## 受け入れ条件` セクション

> **チェックボックス確認**: 読み込んだ時点で `- [x]` と `- [ ]` の数を数えて報告する。
> 例: `完了済み: 2 / 全体: 5 — Task 3 から再開します`

### 過去の決定事項を参照する

他のプランファイルに蓄積された判断を確認し、現在の実装に活かす:

```bash
ls .claude/plans/*.plan.md 2>/dev/null | grep -v "$ARGUMENTS"
```

見つかった各ファイルの `## 実装時メモ` セクションを読む。
現在の機能と関連する判断（避けるべきパターン・技術選択の根拠・既知の落とし穴）を実装の参考にする。

## フェーズ 1 — PREPARE

Git の状態を確認する:

```bash
git branch --show-current
git status --porcelain
```

| 状態 | 対応 |
|------|------|
| `feature/*` ブランチ上 | そのまま続行 |
| `develop` でクリーン | feature ブランチを作成: `git checkout -b feature/{プラン名}` |
| `develop` で未コミット変更あり | **停止** — stash か commit を先に行うよう伝える |
| `main` 上 | **停止** — main への直接コミット禁止 |

## フェーズ 2 — 実装ループ

`## タスクリスト` の `- [ ]` 行を上から順番に処理する。
`- [x]` になっているタスクはスキップする（再開時の継続対応）。

### 各タスク開始時の確認（必須）

**新しいタスクを始める前に、必ずプランファイルを読み直して前タスクが `[x]` になっていることを確認する。**
`- [ ]` のままなら次へ進まず、更新が漏れている原因を調べる。


### TDD タスクの処理

RED / GREEN / REFACTOR フィールドを持つタスクは以下の順で処理する:

#### ステップ A — RED

プランの **RED** フィールドに従い、失敗するテストを書いて実行する。
テストが通ってしまった場合は停止してユーザーに報告する（テスト設計が誤っているか、すでに実装が存在する）。

#### ステップ B — GREEN

プランの **GREEN** フィールドに従い、テストを通す最小限の本番コードを書いて実行する。
- 過剰な実装・最適化・エラーハンドリングを加えない
- GOTCHA に注意する

失敗が残った場合は実装を修正して再実行する（テスト自体は修正しない）。

#### ステップ C — REFACTOR

プランの **REFACTOR** フィールドに従い、テストを緑に保ちながらコードを整える:

- 重複を排除する
- 命名を改善する
- 循環的複雑度 > 10、メソッド長 > 20 行の場合は分割する

リファクタ後にテストを再実行して緑を確認する。

#### ステップ D — チェックボックス更新【必須・省略不可】

> **このステップを飛ばしてはならない。次のタスクへ進む前に必ず実行する。**

Edit ツールを使い、プランファイル（`$ARGUMENTS`）の該当行を更新する:

```
変更前: - [ ] Task N: {タスク名}
変更後: - [x] Task N: {タスク名}
```

更新後、プランファイルを再度読んで `[x]` になっていることを目視確認してから次へ進む。

#### ステップ E — 実装時メモを記録する

**タイミング: タスクの RED〜REFACTOR が完了した直後に記録する。**

Edit ツールでプランファイルの `## 実装時メモ` テーブルに行を追記する:

- 設計上の判断（なぜこのクラス構造にしたか、なぜこのライブラリを使ったか）
- 実装上の注意（特定の書き方をすると Checkstyle / SpotBugs に引っかかる、など）
- 避けたパターンとその理由（試して失敗したアプローチ）

些細な判断は記録しない。将来の自分や他者が「なぜ？」と思いそうなものだけ残す。
実装中ではなく**実装完了後**にまとめて記録することで、思考の流れを妨げない。

## 全タスク完了チェック【ゲート】

> **全タスクが `[x]` になっていない限り、次フェーズに進まない。**

```bash
grep "\- \[ \]" "$ARGUMENTS"
```

出力が空でない場合は未完了タスクがある。該当タスクの実装に戻る。

## フェーズ 3 — 完了報告

```
## 実装完了: {機能名}

プラン   : {パス}
ブランチ : {ブランチ名}
タスク   : {N}/{N} 完了

バリデーション:
  テスト      ✅ {N} テスト

次のステップ:
  - /review {プランファイル} で AI レビューを実施
  - git commit → PR を develop へ作成
```

---

## 失敗時の対応

| 状況 | 対応 |
|------|------|
| RED テストが通ってしまう | 停止 — テスト設計を確認するよう伝える |
| GREEN でテストが失敗し続ける | 実装を修正する（テストは修正しない） |
| チェックボックスの更新を忘れた | 今すぐ Edit ツールで `[x]` に更新してから先へ進む |
| 修正がより多くのエラーを生む | 停止 — ユーザーに報告する |
