## 開発フロー

ブランチ戦略・マージ方針は `Skill('git-workflow')` を参照すること。

設計書・ルールブックを読んだあと、以下の順番で実装を進める。

### ステップ 1 — feature ブランチを切る

```bash
git checkout develop
git pull
git checkout -b feature/<機能名>
```

### ステップ 2 — JUnit TDD（red → green → refactor）

タスクごとに以下のサイクルを回す。

1. **RED** — 失敗するテストを書く（本番コードは書かない）
2. **GREEN** — テストを通す最小限の本番コードを書く
3. **REFACTOR** — テストを緑に保ちながらコードを整える

```bash
./gradlew test --tests "{テストクラス}.{テストメソッド}"
```

- GREEN 未達のままタスクを完了にしない
- リファクタ後も必ずテストを再実行して緑を確認する

----------以下はplan タスクの完了後に行うステップ。planには含めない----------

### ステップ 3 — 静的解析

`reviewer-static-analyst` サブエージェントを起動して静的解析・違反修正を行う。

- Checkstyle・SpotBugs の違反ゼロになってからステップ 4 へ

### ステップ 4 — 仕様書との整合性チェック

`reviewer-implementation-validator` サブエージェントを起動して設計書との整合性を検証する。

- 指摘事項があれば修正し、再度ステップ 3 から実行する
- 指摘なし（または軽微な提案のみ）であればステップ 5 へ

### ステップ 5 — 品質チェック

`reviewer-quality-checker` サブエージェントを起動してセキュリティ・コード品質・パフォーマンスを検証する。

- 指摘事項があれば修正し、再度ステップ 3 から実行する
- 指摘なし（または軽微な提案のみ）であれば完了
